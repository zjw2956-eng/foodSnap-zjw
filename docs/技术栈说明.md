# 技术栈说明

# Flyway — 数据库版本迁移工具。
你把建表 SQL 按版本号命名（如 V1\_\_create_user.sql）放到指定目录，项目启动时 Flyway
会自动按顺序执行未执行过的 SQL。好处是：数据库变更有迹可循，团队协作不用互相传 SQL 脚本，切换环境时能自动重建表结构。

# Shiro — Java 安全框架
负责两件事：认证（验证你是谁，即登录）和授权（你能做什么，即权限控制）。比 Spring Security
轻量，学习门槛低，适合中小项目。

# Realm — Shiro 的概念，可以理解为"数据桥"。
Shiro 不关心你的用户数据存在数据库还是文件里，Realm 负责从你的数据源（如
MySQL）查询用户信息和权限，然后交给 Shiro 做认证授权。你写一个 Realm 类，实现 doGetAuthenticationInfo（登录验证）和
doGetAuthorizationInfo（权限查询）两个方法即可。

# JPA — Java 持久化规范，是一种 ORM 思想（对象-表自动映射）。
本项目同时引入了 JPA 和 MyBatis，实际业务用的都是 MyBatis。你可以暂时忽略
JPA 的部分，你的美食项目用 MyBatis 足够了——SQL 自主可控，学习曲线也更平缓。

# 单点登录（SSO）
核心逻辑

  一句话概括：一个认证中心，多处信任。用户只在 SSO 服务器登录一次，其他子系统通过向 SSO 服务器验证 sessionId 来确认身份，无需重复登录。

  本项目的实现流程（基于 xxl-sso）

  ┌─────────────┐     ①访问资源(未登录)      ┌─────────────┐
  │  用户浏览器   │ ──────────────────────→  │  hope-admin  │
  │              │                            │  (子系统)     │
  └─────────────┘                            └──────┬──────┘
         │                                          │②重定向到SSO登录
         │  ←───────────────────────────────────────┘
         │
         │  ③跳转到SSO登录页(http://127.0.0.1:8887/login)
         ↓
  ┌─────────────┐
  │ hope-sso-   │  ④输入用户名密码
  │   server    │  ⑤验证通过，生成 sessionId（UUID去横杠 + 时间戳）
  │  (认证中心)  │  ⑥sessionId 存入 Redis，同时写入浏览器 Cookie
  └─────────────┘  ⑦带着 sessionId 重定向回子系统
         │
         │  ⑧子系统拿 sessionId 调 SSO 的 /logincheck 接口验证
         ↓
     登录成功，进入子系统

  关键代码在 WebController.doLogin()（hope-sso-server/controller/WebController.java:96）：
  1. 验证用户名密码
  2. 构建 XxlSsoUser 对象，生成 sessionId
  3. 调用 SsoWebLoginHelper.login() 存入 Redis + 写入 Cookie
  4. 将 sessionId 拼接到重定向 URL 返回给子系统

  两条认证通道

  ┌────────────────────────┬───────────────────┬────────────────────┐
  │          通道          │       场景        │ sessionId 传递方式 │
  ├────────────────────────┼───────────────────┼────────────────────┤
  │ Web 端 (WebController) │ 浏览器 B/S        │ Cookie（自动携带） │
  ├────────────────────────┼───────────────────┼────────────────────┤
  │ App 端 (AppController) │ 移动端/前后端分离 │ 请求参数明文传递   │
  └────────────────────────┴───────────────────┴────────────────────┘

  注意事项

  - Redis 是整个 SSO 的命脉 — 所有登录态都存在 Redis 里。Redis 挂了，所有子系统全部无法验证登录
  - SessionId 有效期 — 项目中通过 sso.redis.expire.minite 配置，存在 SsoLoginStore 中
  - Cookie 跨域问题 — SSO 服务器和子系统如果是不同域名，Cookie 默认带不过去，需要同域或额外处理
  - 登出要清 Redis — SsoWebLoginHelper.logout() 同时删除 Redis 中的 session 和浏览器 Cookie，不然会出现"已登出但其他系统仍能访问"的问题

# RBAC "角色-资源"权限模型
 核心逻辑

  一句话：用户 → 角色 → 资源（权限），用户不直接拥有权限，而是通过角色间接获得。这个模型的核心价值在于解耦 — 100
  个用户如果权限相同，只需给一个角色配一次权限，而不是给 100 个用户各配一次。

  数据表结构（5 张表）

  ┌──────────┐     ┌──────────────┐     ┌──────────┐
  │ sys_user │     │ sys_user_role│     │ sys_role │
  │  用户表   │────→│ 用户-角色关联表│←────│  角色表   │
  └──────────┘     └──────────────┘     └────┬─────┘
                                             │
                                      ┌──────↓──────┐
                                      │sys_role_     │
                                      │ resource     │
                                      │ 角色-资源关联表│
                                      └──────┬──────┘
                                             │
                                      ┌──────↓──────┐
                                      │ sys_resource │
                                      │   资源/权限表  │
                                      └──────────────┘

  - sys_user：用户基本信息
  - sys_role：角色（如"管理员""普通用户""编辑"）
  - sys_resource：资源 = 权限点。关键字段 url（接口路径）和 permission（权限标识字符串），type 区分是目录/菜单/按钮
  - sys_user_role：用户和角色的多对多关联
  - sys_role_resource：角色和资源的多对多关联

  Shiro 如何使用这套模型

  看 HopeShiroRealm 中的两个核心方法：

  ① 认证 doGetAuthenticationInfo（hope-core/.../realm/HopeShiroRealm.java:56）
  用户输入用户名密码 → 从数据库查出用户 → 校验账号是否存在/是否被禁用
  → 将密码+盐交给 Shiro 做哈希比对 → 比对通过 = 登录成功
  ② 授权 doGetAuthorizationInfo（hope-core/.../realm/HopeShiroRealm.java:86）
   拿当前登录用户ID → 查 sys_user_role → 得到该用户的所有角色
    → 查 sys_role_resource → 得到该用户的所有权限标识（如 "user:add", "user:delete"）
    → 存入 SimpleAuthorizationInfo → Shiro 自动在访问资源时做拦截比对
   
   注意事项
  - 权限标识是字符串匹配 — sys_resource.permission 字段存储的 user:add 之类的字符串，代码中用 @RequiresPermissions("user:add")
  注解匹配。字符串拼错了权限就不生效
  - Shiro 会缓存授权信息 — 第一次查完后缓存，下次不查数据库。所以修改了用户权限后必须调用 clearAuthorizationByUserId()
  清除缓存（HopeShiroRealm 里已经实现了这个方法）
  - 资源类型分层 — type: 0目录 / 1菜单 / 2按钮，这是前端渲染菜单树的依据，按钮级别的权限控制粒度最细
  - URL 和 Permission 是两套体系 — 项目中 Shiro 过滤器链用 URL 拦截，注解用 Permission 字符串控制。两者配合使用，不要混为一谈


  # 当前项目 CRUD 方式：MyBatis + tk.MyBatis（不是 MyBatis-Plus）

  Mapper → extends tk.mybatis.mapper.common.BaseMapper<T>   (免费CRUD)
  Service → extends BaseService<T>
  ServiceImpl → extends BaseServiceImpl<T>  (默认实现)

  tk.MyBatis（通用 Mapper） 和 MyBatis-Plus 思路相似但不完全一样。它让你不用写 XML 就能获得 insert、selectByPrimaryKey、updateByPrimaryKey、deleteByPrimaryKey、selectAll
  等基础操作。复杂查询才需要自己写 SQL。分页用 PageHelper 插件。

  所以你以后写新功能的 CRUD，套路就是：
  - 建表 → 写实体类 → Mapper 接口继承 BaseMapper<T> → Service 继承 BaseService<T> → Controller 调用。简单表连 XML 都不用写。


 # Python 做 AI Agent，Java 调用

  这个架构完全正确，而且是行业标准做法。 Python 在 AI/ML 生态上无可替代（FastAPI + Transformers + 各种视觉模型），Java 在 Web 业务层稳定可靠。两者通过 HTTP REST API 通信即可——Java 发请求给     
  Python 服务，Python 返回分析结果。

  浏览器 → [Java Spring Boot:8886]  →  [Python FastAPI:5000]
                ↑                            ↓
           业务逻辑+数据库          图片分析/文案生成/建议

  注意两个点：
  - 调用 Python 服务时设置合理的超时时间（AI 推理可能慢，建议 30-60 秒）
  - Python 服务挂了不能影响主流程，做好降级处理（比如返回空、标记"AI 分析中"）


 # Flyway 是什么
     
  一句话：Flyway 是一个数据库版本管理工具，就像 Git 管理代码版本一样，Flyway 管理你的数据库表结构版本。
     
  解决的问题：传统开发中，你在开发环境改了表结构，要手动去测试环境、生产环境也执行一遍 SQL，容易遗漏或执行顺序搞错。Flyway 把这些 SQL
  脚本按版本号管理起来，应用启动时自动检查并执行未执行过的脚本。

  怎么操作：
  1. 把 SQL 文件放到 hope-flyway/src/main/resources/db/migration/ 目录
  2. 文件名必须是 V{版本号}__{描述}.sql（两个下划线）
  3. 启动 hope-flyway 模块（HopeFlywayApplication），它会自动扫描并执行
  4. Flyway 内部用一张 flyway_schema_history 表记录哪些脚本已经执行过，不会重复执行

  铁律：已经执行过的 SQL 脚本绝对不能修改。如果表结构又要改，就新建一个更高版本号的脚本。
