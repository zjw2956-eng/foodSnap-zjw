# 美食分享社交平台 — 架构升级计划

## 一、背景

当前 foodSnap 已完成基础功能开发（发帖、点赞、收藏、评论、弹幕、双榜机制），但架构上仍存在两个层面的问题：

1. **规模层面**：发帖流程是同步的，未来接入 AI 分析后，用户需要等待图片上传 + AI 识别 + 文案生成的全链路，体验极差；点赞/收藏计数采用 DB 全量 `select().size()`，数据量大时性能崩塌。
2. **质量层面**：内容审核只有一层敏感词过滤，无法应对垃圾图片、灌水帖、非食物内容等复杂场景。

本文档基于"解耦与异步化"和"多级质量过滤漏斗"两个核心策略，制定分阶段架构升级计划。

---

## 二、当前架构 vs 目标架构

### 当前（单体同步架构）

```
用户请求 → Controller → Service → Mapper → MySQL
                              ↓
                        同步等待返回
```

- 所有操作同步完成，请求线程被阻塞
- 点赞计数从 DB 全量加载到内存再 `.size()`
- 敏感词每次请求全量查 DB

### 目标（事件驱动 + 分层缓存架构）

```
用户请求 → Controller（薄层，校验 + 返回）
              ↓ 需要异步处理的操作
         RocketMQ ──→ Consumer（AI分析/审核/建索引）
              ↓
    ┌─────────┼─────────┐
    ↓         ↓         ↓
  Minio    MySQL    Elasticsearch
  (图片)   (元数据)   (全文搜索)
              ↓
    ┌─────────┼─────────┐
    ↓         ↓         ↓
  Redis     Redis    Caffeine
  (Set:点赞)(ZSet:榜单)(敏感词)
```

核心变化：
- **请求处理**：同步 → 异步（发帖、AI 分析）
- **互动计数**：DB 查询 → Redis Set 原子操作
- **缓存分层**：Redis 分布式缓存 + Caffeine 本地缓存
- **内容搜索**：SQL LIKE → Elasticsearch 倒排索引
- **内容审核**：单层敏感词 → 规则层 + 模型层 + 人工层 三级漏斗

## 三、升级项详解

### 3.1 互动系统 Redis 化（已完成 ✅）

**解决的问题**：点赞/收藏在高并发下的计数值性能问题。

**升级前**：
```
toggleLike() {
    1. SELECT * FROM food_like WHERE post_id=? AND user_id=?
    2. 如果存在 → DELETE，如果不存在 → INSERT
    3. SELECT * FROM food_like WHERE post_id=? → .size()  ← 全量加载到内存！
    4. UPDATE food_post SET like_count=?
}
```

**升级后**：
```
toggleLike() {
    1. post:like:{postId} → Redis Set<userId>
    2. SISMEMBER 判断是否已点赞 → O(1)
    3. 已赞 → SREM, 未赞 → SADD        → O(1)
    4. SCARD 获取最新计数               → O(1)
    5. 异步：每5分钟定时任务将 Redis 同步到 MySQL（最终一致性）
}
```

**关键技术点**：
- 数据结构选型：Set 天然去重，支持判断成员是否存在
- 原子性：Redis 单线程模型，SADD/SREM/SCARD 天然线程安全
- 持久化策略：Redis 设 7 天 TTL + 定时同步到 DB，兼顾性能和数据安全
- 一致性权衡：最终一致性，允许 Redis 和 DB 有短暂差异（5 分钟内），社交互动可接受

**面试话术**：
> "点赞场景的核心诉求是：判断用户是否已点赞 + 计数。用 Redis Set 的 SISMEMBER + SCARD 都是 O(1)，比 MySQL 的 SELECT COUNT 高效两个数量级。Set 天然去重，不需要额外做幂等判断。通过定时任务同步到 DB 保证最终一致性。"

### 3.2 发帖异步化 — RocketMQ 削峰（下一步）

**解决的问题**：发帖链路过长（上传 → 写DB → AI分析 → 审核），用户等待时间长；午饭/晚饭高峰期瞬时流量大。

**升级前（同步流程）**：
```
用户点击"发布"
  → 上传图片到 Minio（2s）
  → 写入 food_post 表（0.1s）
  → 调用 Python AI 服务识别菜品（3~30s）
  → 生成文案
  → 返回成功

用户等待时间：5~32 秒
```

**升级后（事件驱动异步流程）**：
```
┌─ 阶段一：核心发布（同步，用户感知）─────────────┐
│                                                │
│  用户点击"发布"                                  │
│    → 基本校验（文件大小、格式）                    │
│    → 上传图片到 Minio                           │
│    → 写入 food_post 表（status=2，AI分析中）      │
│    → 发送 MQ 消息"帖子已创建"                     │
│    → 返回成功（帖子ID + "AI分析中"状态）           │
│                                                │
│  用户等待时间：< 3 秒                             │
└────────────────────────────────────────────────┘
                    ↓ 异步解耦
┌─ 阶段二：异步处理（后台，用户无感）───────────────┐
│                                                │
│  RocketMQ 消息"帖子已创建"                       │
│    ├─→ Consumer A：调用 AI 服务                   │
│    │     → 识别菜品名 + 生成文案 + 同款建议         │
│    │     → 写入 food_ai_analysis 表              │
│    │     → 更新 food_post（foodName、description）│
│    │     → 更新 food_post.status = 1（正常）      │
│    │                                            │
│    └─→ Consumer B：内容审核                      │
│          → 图片鉴黄/鉴暴                         │
│          → 文本敏感词检测                        │
│          → 如违规 → 更新 food_post.status = 3     │
└────────────────────────────────────────────────┘
```

**为什么选 RocketMQ 而不是 Kafka？**

| 考虑维度 | RocketMQ | Kafka | foodSnap 选型理由 |
|---------|----------|-------|------------------|
| 延迟消息 | 内置 18 个延迟级别，一行代码 | 不原生支持 | AI 分析需要"用户修改文案"时间窗口 |
| 事务消息 | 阿里双十一验证的分布式事务 | 社区版不够成熟 | 发帖写 DB 和发 MQ 需要原子性 |
| 运维复杂度 | NameServer 轻量 | 依赖 ZooKeeper | 实习生可独立部署 |
| 中文生态 | 极强 | 一般 | 国内面试必问 |
| 吞吐量 | 十万级 TPS | 百万级 TPS | 社交发帖场景十万级够用 |

**面试话术**：
> "发帖是社交平台的核心链路。高峰期（午饭/晚饭时间）大量用户同时发帖，如果同步等待 AI 分析，Tomcat 线程池很快被打满。引入 RocketMQ 后，核心发布路径只做校验 + 落库 + 发消息，毫秒级返回。AI 分析和审核全部异步消费，通过 MQ 实现削峰填谷。选择 RocketMQ 而不是 Kafka，因为延迟消息支持"用户修改文案"的时间窗口，事务消息保证发帖和 MQ 的原子性，避免消息丢失。"

### 3.3 缓存分层策略

#### 3.3.1 榜单 Redis 缓存

**解决的问题**：排行榜是读多写极少场景（30 分钟不变），每次都查 DB 浪费。

**缓存流程（Cache-Aside 模式）**：
```
GET /api/ranking?rankingType=realtime
    →
    1. 查 Redis Key: ranking:realtime:{areaCode}:{category}
       ├─ 命中 → 直接返回
       └─ 未命中 → 查 MySQL food_ranking 表
                    → 写入 Redis（TTL=30分钟，和热榜刷新同步）
                    → 返回

每30分钟热榜刷新时：
    1. 计算新排名
    2. 写入 MySQL
    3. 删除旧 Redis Key（下次查询时重建缓存）
```

**面试话术**：
> "榜单查询用 Cache-Aside 模式。先查 Redis，命中直接返回；未命中查 DB 后回写缓存。TTL 设为 30 分钟和热榜刷新频率一致。刷新时删除缓存而不是更新，避免并发写导致的数据不一致。如果被问到缓存穿透，可以说用布隆过滤器或缓存空值解决。"

#### 3.3.2 敏感词 Caffeine 本地缓存

**解决的问题**：敏感词列表几乎不变，但每次发弹幕/评论都 `SELECT * FROM food_sensitive_word`。

**缓存策略**：
```
发弹幕/评论时：
    1. 从 Caffeine 缓存获取敏感词列表
       ├─ 命中 → 直接用于过滤
       └─ 未命中（首次/过期）→ 查 DB → 写入 Caffeine
    2. 遍历敏感词列表，检查 content 是否包含

管理员新增敏感词时：
    1. INSERT INTO food_sensitive_word
    2. 调用接口刷新 Caffeine 缓存（主动失效）
```

**为什么用 Caffeine 而不是 Redis？**

| 维度 | Caffeine（本地缓存） | Redis（分布式缓存） |
|------|---------------------|-------------------|
| 访问速度 | 纳秒级（JVM 堆内） | 毫秒级（网络 IO） |
| 内存占用 | 每个节点独立占用 | 共享 |
| 一致性 | 更新需要通知所有节点 | 天然一致 |
| 适合场景 | 极少变化、极小数据量 | 需要跨节点共享 |

敏感词列表几百条，几乎不变，Caffeine 是最优解。

**面试话术**：
> "敏感词列表是典型的'读多写极少'场景，用 Caffeine 本地缓存最合适。相比 Redis 省去了网络 IO，纳秒级读取。更新时通过接口主动失效缓存。面试官如果问分布式一致性问题，可以说敏感词场景允许短暂不一致（新增词有几秒延迟才生效），如果要求强一致就改用 Redis。"

### 3.4 内容质量 — 多级过滤漏斗

这是图片 2.2 节的核心思想在 foodSnap 的具体落地。

**当前状态**：只有规则层（敏感词黑名单），拦截维度单一。

**目标架构**：
```
           ┌───────────────────┐
           │   用户发布内容      │
           └────────┬──────────┘
                    ↓
    ┌───────────────────────────────┐
    │  第一层：规则层（成本最低）      │
    │  · 敏感词黑名单                │
    │  · 发布频率限制（1秒内≤3条）     │
    │  · 图片格式/尺寸校验           │
    │  · 空内容/纯表情拦截           │
    │  → 拦截率：~60% 低质内容        │
    └───────────────┬───────────────┘
                    ↓ 通过
    ┌───────────────────────────────┐
    │  第二层：模型层（AI 自动）      │
    │  · Python AI 图片鉴黄/鉴暴     │
    │  · 非食物图片识别              │
    │  · 灌水文本检测（重复字符占比）  │
    │  · 图片质量评估（模糊度/清晰度） │
    │  → 拦截率：~30% 低质内容        │
    └───────────────┬───────────────┘
                    ↓ 低置信度
    ┌───────────────────────────────┐
    │  第三层：人工层（兜底）          │
    │  · AI 置信度<70% 的内容         │
    │  · 用户举报达到阈值             │
    │  · 复杂价值观判断               │
    │  → 分发给审核员，人机协同        │
    │  → 拦截率：~8%                │
    └───────────────┬───────────────┘
                    ↓
              ~98% 拦截率
```

**人机协同机制**：
```
AI 给出审核建议 + 置信度 + 理由
    ├─ 高置信度(>90%) 违规 → 自动下架
    ├─ 中置信度(70-90%) → 推送给审核员，附带 AI 理由
    │   审核员确认违规 → 下架 + 标记为训练样本
    │   审核员判定正常 → 放行 + 反馈给模型
    └─ 低置信度(<70%) → 默认放行，事后抽样复审
```

**面试话术**：
> "内容审核不能只靠关键词过滤。我们设计了规则层 → 模型层 → 人工层的三级漏斗。规则层挡掉60%的明显垃圾，模型层用AI处理复杂情况（鉴黄、食物识别），最后人工兜底边缘案例。关键是'人机协同'——AI给出置信度和理由，审核员确认或驳回，驳回的case反馈给模型做持续训练。"

### 3.5 远期规划

| 升级项 | 说明 | 技术选型 |
|--------|------|---------|
| 全文搜索 | 帖子搜索、菜品名模糊匹配 | Elasticsearch + IK 中文分词 |

## 五、面试核心叙事

把以上技术点串成一条面试故事线：

> "这个项目的架构演进分三个阶段。第一阶段完成基础功能——发帖、互动、双榜。第二阶段解决性能瓶颈——点赞/收藏用 Redis Set 替代 DB 查询实现 O(1) 操作；发帖接入 RocketMQ 实现异步削峰，用户发帖 3 秒内返回，AI 分析和审核后台消费；榜单加 Redis 缓存，敏感词用 Caffeine 本地缓存。第三阶段侧重内容质量和检索——Elasticsearch 做全文搜索，三级漏斗做内容审核，AI 模型持续优化识别准确率。"
>
> "核心思想是图片里提到的两个：规模上用'解耦与异步化'应对高并发，质量上用'多级过滤漏斗'净化内容生态。"
